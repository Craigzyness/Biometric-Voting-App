# Backend-Managed Proxy Voting: Address Pool Design

## 1. Introduction
To enhance voter anonymity beyond using a single backend signer address for all on-chain votes, we will implement a backend-managed proxy voting system. This system will utilize a pool of pre-funded Ethereum addresses to submit vote transactions to the `ElectionManager` smart contract. This document outlines the design for managing this address pool.

The primary goal is to make it harder to link votes on the blockchain to a single originating entity and, by extension, to individual application users (though application-level anonymity and unlinkability to the *actual voter* still relies on the backend's integrity).

## 2. Design Components

### 2.1. Address & Private Key Management

#### 2.1.1. Generation/Acquisition
-   **For PoC & Local Development (Ganache):**
    -   Ganache automatically generates a list of accounts with private keys upon startup. We can directly use a subset of these private keys.
    -   Alternatively, new accounts can be generated using `web3.py` or other Ethereum libraries, and their private keys recorded.
-   **For Staging/Production (Future):**
    -   Dedicated, secure generation process for new private keys.
    -   Consider Hierarchical Deterministic (HD) wallets to manage a tree of keys from a single seed, but individual private keys will still need to be used by the backend.

#### 2.1.2. Secure Storage (PoC vs. Production)
-   **PoC (Local Development):**
    -   **Option A (Simplest):** Store a list of private keys directly in an environment variable (e.g., `GANACHE_POOL_PRIVATE_KEYS` as a comma-separated string). This is easy for local setup but not secure for shared environments.
    -   **Option B (Slightly Better PoC):** Store private keys in a JSON file (e.g., `pool_keys.json`) that is:
        -   **Strictly .gitignored.**
        -   Loaded by the `blockchain_service.py` at startup.
        -   The file itself should have restricted permissions on the filesystem.
    -   This document will assume **Option B (JSON file)** for the PoC implementation steps due to its manageability for a small pool of keys.
-   **Staging/Production (Future):**
    -   **HashiCorp Vault:** Recommended for managing secrets like private keys. The backend application would authenticate to Vault to retrieve keys as needed.
    -   Encrypted database storage with strict access controls and KMS integration.
    -   Hardware Security Modules (HSMs) for ultimate private key protection (most complex and costly).

### 2.2. Address Funding (Gas)
-   **PoC (Ganache):**
    -   Accounts generated by Ganache are pre-funded with ample ETH. This is straightforward.
    -   If generating new accounts for the pool, they will need to be manually funded by transferring ETH from one of Ganache's default funded accounts.
-   **Staging/Production (Future):**
    -   A robust system for monitoring pool address balances and automatically topping them up from a central treasury wallet will be required. This is a significant operational component.

### 2.3. Address Selection Logic
When a vote needs to be cast on-chain:
1.  **Retrieve Available Pool:** The backend will have a list of (address, private_key) pairs loaded.
2.  **Selection Strategy:**
    -   **Option A (Round-Robin):** Cycle through the pool addresses sequentially. Simple to implement.
    -   **Option B (Random Selection):** Randomly pick an address from the pool.
    -   **Option C (Least Recently Used - LRU):** Maintain usage timestamps or a queue to pick the address that hasn't sent a transaction the longest. This helps distribute activity.
    -   **For PoC:** Round-robin or random selection is sufficient. Let's start with **Round-Robin**.
3.  **Concurrency:** If the backend handles multiple vote requests concurrently, the selection logic must be thread-safe to avoid multiple requests trying to use the same address for different transactions simultaneously before its nonce is updated. A locking mechanism or a queue for addresses might be needed for high concurrency. (For PoC, assume serial processing or low concurrency where simple iteration is okay).

### 2.4. Address Usage Tracking & Cooldown (Optional for Initial PoC)
-   **Goal:** To avoid using the same pool address too frequently, which could create linkable patterns if an adversary is monitoring the pool.
-   **Mechanism (Conceptual):**
    -   Maintain a timestamp for when each pool address last submitted a transaction.
    -   When selecting an address, prefer those that haven't been used recently or have a "cooldown" period.
    -   This adds complexity to the selection logic.
-   **For Initial PoC:** We will omit explicit cooldown tracking beyond the round-robin selection to keep it simple. The primary goal is to use different addresses than a single backend master address.

### 2.5. Error Handling
-   **Pool Exhaustion:**
    -   What if all available addresses are temporarily "in use" (e.g., waiting for a transaction to be mined, if we implement a check to use an address only once at a time before its nonce is confirmed for the next tx)?
    -   For PoC with round-robin, this is less of an issue unless the pool is tiny and all are simultaneously processing.
    -   **Action:** Log the error. The API might return a temporary server error (e.g., 503 Service Unavailable) asking the user to try again shortly.
-   **Insufficient Gas on a Pool Address:**
    -   The transaction will fail.
    -   **Action:** The `blockchain_service` should catch this (if possible from the error type). The address should ideally be marked as "needs funding" and temporarily removed from the active selection pool. A notification system for ops would be needed in production. For PoC, log the error.
-   **Private Key Compromise:** This is a security incident. In production, a mechanism to revoke/disable a compromised pool address would be needed.

## 4. Impact on Smart Contract
-   The `ElectionManager.sol` contract's `voterHasVoted` mapping (`mapping(uint256 => mapping(address => bool))`) will now effectively store the pool addresses that have cast votes for a given election.
-   The smart contract itself does not need changes to support this proxy system, as it simply sees `msg.sender` as the address submitting the transaction.

## 5. Security Considerations for This Approach
-   **Trust in Backend:** This scheme shifts, but does not eliminate, trust. Users trust the backend to:
    -   Not link their application identity to the specific proxy address used for their vote.
    -   Manage the pool addresses and their keys securely.
    -   Select proxy addresses in a way that enhances unlinkability.
-   **Backend Security:** The security of the backend server and its key management practices becomes paramount.
-   **Pool Size & Management:** A small or poorly managed pool can offer weak anonymity. The larger and more actively/randomly used the pool, the better.

This design provides a pragmatic first step towards on-chain vote anonymity by breaking the direct link from a single backend signer.
