# Database Schema Design

Date: (Refer to Git history for the last update date of this document)

This document outlines the proposed database schema for the Biometric Voting App. It assumes a relational database model (e.g., PostgreSQL) based on the considerations in `01_technology_stack_considerations.md`.

## Table Definitions:

### 1. `Voters` Table

Stores information about registered voters, identified by their app-generated anonymized ID.

*   **Purpose:** To maintain a record of unique, registered (anonymized) voters and their eligibility.
*   **Schema:**
    *   `id`:
        *   Data Type: `UUID` (Primary Key) or `BIGSERIAL` (auto-incrementing 64-bit integer, Primary Key)
        *   Constraints: Not Null, Primary Key
        *   Description: Unique internal identifier for the voter record.
    *   `anonymized_voter_id`:
        *   Data Type: `VARCHAR(255)` (e.g., for SHA-256 hex string)
        *   Constraints: Not Null, Unique
        *   Description: The anonymized ID generated by the client application. This is the public-facing identifier for a voter.
        *   Index: Yes (Unique Index)
    *   `registration_timestamp`:
        *   Data Type: `TIMESTAMP WITH TIME ZONE`
        *   Constraints: Not Null, Default: `CURRENT_TIMESTAMP`
        *   Description: Timestamp of when the voter was registered.
    *   `is_eligible`:
        *   Data Type: `BOOLEAN`
        *   Constraints: Not Null, Default: `TRUE`
        *   Description: Flag to indicate if the voter is currently eligible to participate in elections. Can be used for administrative purposes (e.g., deactivating a compromised ID if such a mechanism is ever needed, though complex for anonymity).
    *   `created_at`:
        *   Data Type: `TIMESTAMP WITH TIME ZONE`
        *   Constraints: Not Null, Default: `CURRENT_TIMESTAMP`
    *   `updated_at`:
        *   Data Type: `TIMESTAMP WITH TIME ZONE`
        *   Constraints: Not Null, Default: `CURRENT_TIMESTAMP` (should auto-update on record change if DB supports it)

### 2. `Elections` Table

Stores details for each election event.

*   **Purpose:** To define available elections, their properties, options, and active periods.
*   **Schema:**
    *   `id`:
        *   Data Type: `UUID` (Primary Key) or `BIGSERIAL` (Primary Key)
        *   Constraints: Not Null, Primary Key
        *   Description: Unique internal identifier for the election.
    *   `election_code`:
        *   Data Type: `VARCHAR(64)`
        *   Constraints: Not Null, Unique
        *   Description: A human-readable, unique code for the election (e.g., "PRES2024", "PROP-CITY-PARK-01").
        *   Index: Yes (Unique Index)
    *   `title`:
        *   Data Type: `VARCHAR(255)`
        *   Constraints: Not Null
        *   Description: The official title of the election.
    *   `description`:
        *   Data Type: `TEXT`
        *   Constraints: Nullable
        *   Description: A more detailed description of the election or proposition.
    *   `options`:
        *   Data Type: `JSONB` (PostgreSQL) or `TEXT` (storing a JSON string). `TEXT[]` (array of strings) is also an option in PostgreSQL.
        *   Constraints: Not Null
        *   Description: An ordered list of the voteable options for this election. Example: `["Candidate A", "Candidate B", "Abstain"]` or `[{"id": "opt1", "text": "Option 1"}, {"id": "opt2", "text": "Option 2"}]`. JSONB is flexible.
    *   `start_timestamp`:
        *   Data Type: `TIMESTAMP WITH TIME ZONE`
        *   Constraints: Not Null
        *   Description: The exact date and time when voting for this election begins.
        *   Index: Yes
    *   `end_timestamp`:
        *   Data Type: `TIMESTAMP WITH TIME ZONE`
        *   Constraints: Not Null
        *   Description: The exact date and time when voting for this election ends.
        *   Index: Yes
    *   `status`:
        *   Data Type: `VARCHAR(20)` (e.g., 'PENDING', 'ACTIVE', 'CLOSED', 'ARCHIVED')
        *   Constraints: Not Null, Default: 'PENDING'
        *   Description: Current status of the election. Could be managed by scheduled jobs based on start/end timestamps.
    *   `created_at`:
        *   Data Type: `TIMESTAMP WITH TIME ZONE`
        *   Constraints: Not Null, Default: `CURRENT_TIMESTAMP`
    *   `updated_at`:
        *   Data Type: `TIMESTAMP WITH TIME ZONE`
        *   Constraints: Not Null, Default: `CURRENT_TIMESTAMP`
    *   **Additional Constraints:**
        *   `CONSTRAINT check_election_dates CHECK (start_timestamp < end_timestamp)`: Ensures the election start time is before its end time.

### 3. `Votes` Table

Records each vote cast by a voter for a specific election.

*   **Purpose:** To securely store individual votes, ensuring integrity and preventing double voting.
*   **Schema:**
    *   `id`:
        *   Data Type: `UUID` (Primary Key) or `BIGSERIAL` (Primary Key)
        *   Constraints: Not Null, Primary Key
        *   Description: Unique internal identifier for the vote record.
    *   `voter_id`:
        *   Data Type: `UUID` or `BIGINT` (matches `Voters.id` type)
        *   Constraints: Not Null, Foreign Key (references `Voters(id)`)
        *   Description: Links to the voter who cast this vote.
    *   `election_id`:
        *   Data Type: `UUID` or `BIGINT` (matches `Elections.id` type)
        *   Constraints: Not Null, Foreign Key (references `Elections(id)`)
        *   Description: Links to the election this vote is for.
    *   `selected_option_value` (or `selected_option_index` or `selected_option_hash`):
        *   Data Type: `TEXT` (if storing the option text directly, or a hash) or `INTEGER` (if storing the index of the option from `Elections.options` array).
        *   Constraints: Not Null
        *   Description: The actual choice made by the voter.
            *   Storing hash of (option text + election_id + potentially voter_id or a per-vote salt) could enhance privacy if the raw selected option text is sensitive and needs to be obscured even in the DB, while still allowing for tallying if hashes are consistent.
            *   Storing an index is simple if options are always presented in the same order.
            *   For MVP, storing the selected option text might be simplest, with hashing considered for future privacy hardening.
    *   `encrypted_proof`:
        *   Data Type: `TEXT`
        *   Constraints: Nullable
        *   Description: Stores the encrypted proof payload from the client, associated with the vote. This proof is generated by the client using a biometric-bound key and contains details of the vote, providing a cryptographic link between the biometric action and the vote submission.
    *   `iv`:
        *   Data Type: `TEXT`
        *   Constraints: Nullable
        *   Description: Stores the Initialization Vector (IV) used for encrypting the `encrypted_proof`. Necessary for decryption if the proof needs to be verified.
    *   `cast_at_timestamp`:
        *   Data Type: `TIMESTAMP WITH TIME ZONE`
        *   Constraints: Not Null, Default: `CURRENT_TIMESTAMP`
        *   Description: Timestamp of when the vote was successfully recorded by the backend.
    *   `is_valid`:
        *   Data Type: `BOOLEAN`
        *   Constraints: Not Null, Default: `TRUE`
        *   Description: Flag to mark if the vote is considered valid during tallying (e.g., could be set to false if some anomaly is detected post-casting, though this requires careful rules).

*   **Unique Constraints:**
    *   A composite unique constraint on (`voter_id`, `election_id`) is **CRITICAL** to prevent a voter from casting more than one vote in the same election at the database level.

*   **Indexes:**
    *   Index on `voter_id`.
    *   Index on `election_id`.
    *   Composite index on (`voter_id`, `election_id`) for the unique constraint and for efficient lookup.

## Relationships:

*   A `Voter` can have many `Votes`.
*   An `Election` can have many `Votes`.
*   Each `Vote` belongs to exactly one `Voter` and one `Election`.

This schema provides a foundational structure for a persistent backend. Further details like specific indexing strategies beyond unique keys would be refined based on query patterns and performance testing.
